# BuckyOS Users & Agents 产品原型设计

## 1. 文档说明

- **文档名称**：BuckyOS Users & Agents 产品原型设计
- **产品名称**：Users & Agents
- **技术域名称**：DID 管理 / 实体管理 / 联系人管理
- **文档目标**：基于语音记录，整理出可用于产品评审、交互设计和原型制作的统一方案
- **适用范围**：BuckyOS 桌面端优先，兼顾移动端后续简化适配
- **关联模块**：MessageHub、DID Document、Profile、Message Tunnel Binding、权限/分组系统、Agent 运行管理

---

## 2. 背景与产品定位

BuckyOS 当前需要一个高频使用的“用户与实体管理”组件。它不是传统意义上单纯的“操作系统用户管理”，也不是单纯的“通讯录”或“IAM 联系人管理”，而是三者的融合：

1. **操作系统用户管理**：支持在当前系统空间内创建可登录的新用户；
2. **联系人与群组管理**：承接用户日常的联系人、分组、导入、合并与来源管理；
3. **DID 实体管理**：统一承载用户、Agent、群组等实体的 DID / Profile / Document / Binding 信息。

由于系统底层复用了同一套分组和权限逻辑，产品层不能把这些能力分拆得过于“工程化”，否则普通用户会感到理解成本过高。因此，本组件应采用更贴近 IM/通讯录心智的产品表达，将复杂的 DID、权限、绑定等机制隐藏在一致的交互框架中。

**结论**：对外产品名称建议采用 **Users & Agents**。  
它既能覆盖“用户”与“联系人”，又能明确传达 Agent 在系统中与用户并列的重要地位。

---

## 3. 产品目标

### 3.1 目标

本组件需要同时满足以下目标：

- 对普通用户足够友好，看起来像一个易用的“用户/联系人管理器”；
- 能承载系统级权限、分组、实体身份等关键能力；
- 让“自己”“Agent”“本空间用户”“联系人”“群组”在同一界面中被统一理解；
- 成为 MessageHub、HomeStation 等系统级模块可复用的实体信息入口；
- 为未来的 DID 联合登录、可信绑定、AI 辅助整理预留结构。

### 3.2 设计原则

1. **用通讯录心智承载 DID 能力**  
   用户先看到的是“人、Agent、群组”，不是“身份协议”。

2. **高频功能前置，低频复杂功能后置**  
   搜索、查看、导入、绑定、修改个人信息应显著可达；合并、手工整理等低频操作尽量收敛。

3. **统一实体视图，不强迫用户理解底层对象类型**  
   一个实体可能同时出现在多个组中，但在用户视角仍应是“同一个对象”。

4. **与 MessageHub 高度协同，但职责分明**  
   Users & Agents 更偏结构化管理、导入与整理；MessageHub 更偏即时互动、好友请求、单次添加和会话。

5. **安全优先**  
   尤其对于未来“外部一级 DID 登录本系统”的场景，绝不能让用户在当前系统中输入其外部身份体系的原始密码。

## 4. V1 产品范围

### 4.1 本期必须支持

- 左侧树形实体侧边栏；
- “自己 / Agent / 本空间用户 / 联系人 / 群组”的统一展示；
- 新建本空间用户（二级 DID 用户）向导；
- 自己与 Agent 的详情页；
- Binding 管理入口；
- Profile 信息与 DID Document 的分层编辑；
- 联系人导入能力；
- 联系人来源、来源历史、同步来源展示；
- 分组整理、删除等管理能力；
- 群组/集合的基础展示；
- 桌面端原型优先。

### 4.2 本期保留入口但可不完整实现

- 使用外部一级 DID 联合登录到当前系统；
- 深度的可信绑定验证；
- 自动化联系人合并策略；
- 多账号快速切换；
- 去中心化多方共同 Host 群组；
- 移动端复杂整理操作。

## 5. 核心概念模型

### 5.1 实体分类

Users & Agents 中需要统一表达以下对象：

#### A. 一级对象（系统首页级对象）
默认至少包含两个：

- **自己（Self）**
- **Agent**

这两个对象都应拥有 DID，并在左侧列表中以更高权重展示。

#### B. 本空间用户（Local User）
指在当前系统空间内新增、可实际登录系统的用户。  
本期主要通过创建 **二级 DID** 的方式生成。

#### C. 联系人（Contact）
用户维护的联系人条目，既可能是原生 DID 联系人，也可能来自导入或外部同步源。

#### D. 群组（Group）
需要区分两种不同的“组”：

1. **集合型组（Collection / Tag Group）**  
   本质是对一组对象的逻辑集合，不产生新的独立实体。

2. **实体型组（Entity Group / Physical Group）**  
   真实存在、可被 MessageHub 作为消息对象处理的群组；该类对象可拥有自己的 DID、Owner 和详情页。

---

### 5.2 DID 层级

#### 一级 DID
- 独立存在于当前系统空间之外；
- 可理解为全局身份；
- 示例：`Alice`

#### 二级 DID
- 依附于某个系统空间存在；
- 当前空间销毁后，该身份随之消失；
- 常用于本空间内新增用户或 Agent；
- 示例：`Bob.Alice`

### 5.3 Binding

Binding 即 **Message Tunnel Binding**，表示将某个现实世界中的消息通道账号绑定到本系统实体上，例如 Telegram。

Binding 是 MessageHub 变得“有内容”的关键入口。  
没有 Binding，用户只能在本系统内部实体之间通信；有 Binding 后，系统才具备聚合外部消息网络的能力。

### 5.4 Profile 与 DID Document

用户公开信息分为两层：

#### Info（轻量公开资料）
- 存放于标准 Profile 路径；
- 例如头像、公开简介等；
- 修改成本低，立即生效。

#### DID Document（可信身份资料）
- 承载更严肃、可信的身份信息；
- 未来可能涉及链上或空间外写入；
- 修改可能存在手续费或额外确认成本；
- 例如正式姓名、公钥等。

这两层在 UI 上必须明显区分，避免用户把所有资料都当成“随手可改的个人资料”。

## 6. 产品信息架构

### 6.1 整体结构

页面采用典型的桌面端双栏结构：

```text
┌────────────────────────────────────────────────────────────┐
│ Users & Agents                                            │
├────────────────────────────────────────────────────────────┤
│ 左侧：实体侧边栏              │ 右侧：详情/管理视图         │
│ - 搜索框                      │ - 选中对象的详情页          │
│ - Type / Tag 过滤             │ - 编辑、绑定、导入、整理     │
│ - 可展开树形结构              │ - 向导、操作入口、来源信息   │
│ - 一级对象 / 用户 / 联系人 /组 │                            │
└────────────────────────────────────────────────────────────┘
```

### 6.2 左侧侧边栏

左侧是本组件最核心的导航区，整体风格应与 MessageHub 的实体列表高度相似，以降低学习成本。

结构建议如下：

1. **顶部搜索框**
   - 快速定位某个实体；
   - 搜索应覆盖名称、别名、备注、DID、来源后缀等。

2. **Type / Tag 过滤器**
   - 按对象类型过滤：用户、Agent、联系人、群组等；
   - 按 Tag/标签过滤：仅查看被标记对象。

3. **树形列表**
   - 支持展开/收起；
   - 能表达“分组包含关系”；
   - 同一实体可以在多个分组中重复出现；
   - 这里的“重复出现”是逻辑视图，不意味着重复存储。

### 6.3 列表项表现规则

#### 两行卡片
用于高权重、可管理实体，例如：

- 自己
- Agent
- 自己创建或持有的实体型群组
- 本空间内重点可管理对象

表现建议：
- 清晰头像/Logo
- 主名称 + 副信息
- 让用户一眼感知“这是系统中的重要对象”

#### 单行列表项
用于普通联系人或次级对象。

表现建议：
- 名称简洁
- 可带来源后缀或状态标识
- 用于高密度滚动浏览

### 6.4 主视图区

右侧为选中对象后的详情和操作区域。  
不建议采用“左侧选组、右侧只列组成员”的传统后台式双重管理模式，而应统一使用“选中对象 -> 查看该对象完整详情”的模式。

## 7. 页面与对象详情设计

### 7.1 自己（Self）详情页

自己是默认最高频对象之一，建议结构如下：

### 页面结构建议

1. **头部信息**
   - 头像
   - 显示名
   - DID
   - 用户类型/身份标签

2. **Binding（置顶）**
   - 当前已绑定的 Message Tunnel 账号
   - 支持新增、查看状态、解除绑定
   - 这里是理解 Message Tunnel 的关键入口

3. **Info**
   - 头像
   - 昵称
   - 简介
   - 其他轻量公开资料
   - 修改后立即生效

4. **DID Document**
   - 展示可信身份信息
   - 点击“修改”后进入向导
   - 需要明确提示：可能产生额外确认或成本

5. **安全与账号**
   - 修改密码
   - 重置密码
   - 查看账号状态

### 设计重点

- Binding 的优先级应高于普通 Profile；
- Document 与 Info 必须分区；
- 修改 Document 不应是“就地编辑”，应通过向导完成。

---

### 7.2 Agent 详情页

Agent 与用户在语义上应被设计为“并列的重要对象”，这是 BuckyOS AI Native 心智的重要入口。

### Agent 与用户的共同点

- 有 DID
- 有 Profile
- 有 DID Document
- 有 Binding

### Agent 与用户的差异点

- **没有密码**
- 需要额外展示运行态信息

### Agent 页面建议分区

1. **头部信息**
   - Agent 名称
   - Logo / Avatar
   - DID
   - 所属关系

2. **Binding**
   - 对 Agent 尤其关键；
   - 决定用户是否能通过外部消息通道与 Agent 交互。

3. **Info**
   - 展示对外描述、能力摘要、头像等

4. **DID Document**
   - 可信身份与配置入口

5. **运行与工作信息**
   - 最近工作日志
   - UI Session / Work Session 数量
   - Workspace 数量
   - 跳转到 Agent 工作目录或专属管理页

### 特别说明：Binding 对 Agent 的重要性

- 如果用户自己未绑定 Telegram，但 Agent 绑定了 Telegram：  
  用户从 Telegram 与 Bot 通信时，系统可能无法识别“这个 Telegram 账号就是 Agent 的主人”。

- 如果用户绑定了 Telegram，但 Agent 没绑定：  
  用户无法在标准 Telegram 客户端中直接看到 Agent，只能通过 MessageHub 与其交互。

---

### 7.3 本空间用户详情页

本空间用户是“真正可登录系统”的用户对象，本期主要通过创建二级 DID 用户实现。

页面建议包含：

- 头像 / 名称 / DID
- 用户类型
- 初始账号信息状态
- 可用应用列表
- 所属默认组
- 密码管理
- Binding（如允许）
- Profile / Document

### 默认组约束

当前系统空间中的所有本空间用户，应默认属于同一个不可移除的基础组。  
原因在于系统内存在共享资源（例如公共共享目录），需要一个稳定的基础权限边界。

---

### 7.4 联系人详情页

联系人详情页更偏“聚合资料视图”，而不是系统账号管理页。

建议包含：

1. **基础身份**
   - 名称
   - 备注
   - 来源标识
   - DID（如有）
   - 是否为双向关系

2. **来源信息**
   - 导入来源
   - 最近同步来源
   - 历史导入批次
   - 创建时间 / 最近更新时间

3. **Binding / Reachability**
   - 此联系人有哪些已知可达通道
   - 例如 DID、Telegram、Email 等
   - 未来可展示为“可达会话列表”

4. **组与标签**
   - 所属集合
   - 所属群组
   - 是否在某个特定业务组中

5. **整理操作**
   - 删除
   - 调整分组
   - 合并到其他联系人
   - 交给 Agent 自动整理（未来）

### 来源后缀展示建议

对于从外部系统同步来的名字，可在视图层使用来源后缀，例如：

- `Bob.telegram`
- `Alice.imported`

其作用不是长期暴露“脏数据”，而是帮助用户理解来源、避免命名冲突，并为后续合并提供依据。

---

### 7.5 群组详情页

群组详情页需要支持两类对象：

#### A. 集合型组
- 本质是逻辑集合；
- 用于筛选、标签、可见性控制；
- 不一定可被 MessageHub 直接发消息。

#### B. 实体型组
- 有独立实体身份；
- 可被宿主系统 Host；
- 可作为 MessageHub 中的消息对象；
- 拥有成员、Owner、DID 和自身详情页。

### 页面建议内容

- 群组名称
- 群组类型
- Owner / Host 信息
- 群组 DID（如有）
- 成员列表
- 标签与描述
- 是否可在 MessageHub 发消息
- 与权限系统的关联说明

---

## 8. 关键用户流程

### 8.1 新建本空间用户（主流程）

这是本模块最关键的系统级能力之一。

### 触发入口
- 左侧侧边栏顶部“+”
- 或主视图区显式“添加用户”

### 向导流程

#### 第一步：说明操作性质
必须明确告诉用户：

- 这是在当前系统空间中创建一个“真实可登录”的新用户；
- 该用户会占用系统资源；
- 如果你的目标只是添加联系人，不应该使用这个入口。

#### 第二步：选择用户来源
建议原型中保留两个选项：

1. **创建新用户（本期支持）**
   - 创建二级 DID 用户

2. **使用已有一级 DID 加入（本期预留）**
   - 未来采用联合登录/凭证方式
   - 本期可灰置或标注“即将支持”

#### 第三步：填写基础信息
- 用户名
- 用户类型
- 初始密码

#### 第四步：确认应用可用性
本期建议以“信息展示”为主：

- 当前有哪些应用默认对该用户可用；
- 当前版本通常由 Owner 统一安装应用；
- 后续添加用户暂不开放自主安装应用。

#### 第五步：完成创建
- 自动加入默认基础组；
- 完成后可进入该用户详情页继续设置。

### 交互要点

- 整个流程必须“像向导”，不能直接抛给用户一堆账号字段；
- 重点是让普通用户知道“这是系统用户，不是联系人”。

---

### 8.2 外部一级 DID 登录（预留流程）

该流程在概念上非常重要，但本期可仅保留入口和交互占位。

### 目标形态
用户不需要在新系统中重新注册账号，而是使用自己已有的一级 DID 加入某个组织或系统空间。

### 安全原则
**绝不能要求用户在当前系统输入其外部账号的原始密码。**

理想方案应为：

- 用户先在自己的身份体系中完成认证；
- 由其 DID Document 对应的认证公钥签发凭证（如 JWT）；
- 当前系统只验证凭证，不接触原始密码。

### 原型建议
- 保留入口；
- 文案明确“未来将支持安全联合登录”；
- 不在本期实现完整登录过程。

---

### 8.3 修改个人信息

### Info 修改
- 直接编辑；
- 即时生效；
- 成本低。

### DID Document 修改
- 进入单独向导；
- 显示风险/成本提示；
- 明确告诉用户：这是更正式、可信的身份资料。

---

### 8.4 添加/管理 Binding

Binding 是用户首次理解“Message Tunnel”概念的重要入口。

### 页面要求

- 展示当前支持的 Binding 类型；
- 说明该 Binding 将带来什么能力；
- 清楚展示绑定状态、最近同步时间、异常状态。

### 价值说明
例如绑定 Telegram 后：

- 可以把 Telegram 中的大量消息与联系人同步进系统；
- MessageHub 将拥有更完整的消息上下文；
- 联系人聚合能力显著提升。

---

### 8.5 联系人导入（核心流程）

导入是联系人系统的主入口，优先级高于手工逐个添加。

### 原则

- 用户今天不太可能从零开始重建联系人体系；
- 大多数联系人来自历史系统迁移或外部同步；
- 导入过程必须尽量“安全、可理解、不覆盖旧数据”。

### 支持方式建议

#### 桌面端
- CSV 导入
- XML 导入
- 其他常见通讯录导出格式

#### 移动端（后续）
- 导入系统通讯录
- 从手机通讯录读取手机号等信息

### 导入后需要处理的问题

- 新导入联系人如何与历史联系人整合；
- 是否自动识别重复项；
- 是否产生“影子联系人”或候选合并项；
- 是否保留来源追踪；
- 如何保证“不会随便覆盖现有资料”。

### 联系人来源视图

系统应为用户提供一个总览入口，至少能看到：

- 联系人总数
- 群组数量
- 各来源数量
- 导入历史批次
- 哪些数据来自同步源（如 Telegram）
- 哪些数据来自手工导入

---

### 8.6 手工添加联系人（产品边界建议）

手工添加联系人仍然需要支持，但建议在职责上与 MessageHub 分工：

### Users & Agents 负责
- 导入
- 结构化整理
- 来源管理
- 分组与删除
- 查看详情

### MessageHub 负责
- 单次添加 DID 联系人
- 好友请求
- 双向状态建立
- 消息通知与过滤

### 原因
单个联系人添加往往伴随：

- 是否双向建立关系
- 对方是否接受
- 通知投递
- 垃圾消息过滤

这些更符合 MessageHub 的职责边界。

### 本组件中的表现建议

在 Users & Agents 中可保留“添加联系人”入口，但更建议引导至：

- “从 MessageHub 添加”
- “从导入添加”
- “从绑定来源同步”

---

## 9. 联系人关系与状态设计

### 9.1 DID 联系人的单向 / 双向关系

原生 DID 联系人不是“知道 DID 就自动互为好友”。

### 单向关系
- 我添加了对方；
- 对方未必添加我；
- 我可以读取其公开资料；
- 我给对方发消息，对方未必能看到或愿意接收。

### 双向关系
- 双方都已确认彼此；
- 可以放心地双向通信；
- 是更接近传统“好友”的状态。

这一状态模型尤其适合去中心化体系，能够降低垃圾消息问题。

---

### 9.2 外部通道联系人

对于 Telegram、Email 等外部通道联系人，产品逻辑应尽量遵循来源系统本身的关系模型。

例如：

- 如果用户已经绑定 Telegram，那么 Telegram 联系人应主要通过 Telegram 同步得到；
- 不建议在 Users & Agents 中单独提供“手输 Telegram 账号直接加好友”的主入口；
- 但可以允许用户把 Telegram 身份作为单向记录加入联系人体系中。

---

### 9.3 通过手机号 / 邮箱发现 DID（未来能力）

系统未来可以支持“已知手机号/邮箱，但不知道 DID”时的发现流程。

### 基本思路
- 用户本地拥有对方手机号/邮箱；
- 系统基于标准规则计算哈希；
- 去公开系统中匹配 DID Document 或相关索引；
- 若匹配成功，自动补全对方 DID。

### 产品注意事项
- 这是“发现 DID”的能力，不等于自动建立双向关系；
- 需要明确隐私提示；
- 可信度与匹配置信度应可视化。

---

## 10. 分组、标签与显示逻辑

### 10.1 同一实体可重复出现在多个组中

这是本组件的重要特征：

- 同一个实体可以属于多个逻辑组；
- 在左侧树中可多处出现；
- 这是视图复用，不是实体复制。

### 10.2 分组不等于权限组，但可映射

虽然底层权限系统与分组系统有较深关联，但产品层不应直接把它表达为“权限树管理器”。

对用户而言，分组更像：

- 通讯录分组
- 标签集合
- 群组
- 可见性范围

在需要时，再由系统把这些结构映射到底层权限能力。

### 10.3 自建组与加入的组

对于实体型组，需要区分：

### 自己创建/Host 的组
- 属于高权重对象；
- 可采用两行卡片表现；
- 与“自己”“Agent”处于相近层级。

### 自己加入的组
- 更像联系人对象；
- 可带特殊标识；
- 出现在联系人体系中。

---

## 11. 低频管理操作设计

### 11.1 备注
备注是高频能力，但很可能在 MessageHub 中更常发生。  
因此建议将“备注编辑”设计为系统通用组件，Users & Agents 和 MessageHub 共用。

### 11.2 删除
删除是 Users & Agents 中相对合理的高频整理操作。

适用场景：

- 清理错误导入；
- 移除无效联系人；
- 删除临时分组。

### 11.3 调整分组
这是典型的结构化整理行为，更适合放在 Users & Agents 中完成。

### 11.4 手工合并
手工合并是必要但低频、且交互复杂的能力。

### 桌面端建议方案
优先考虑以下方式：

- **多选后右键合并**
- 弹出确认框
- 选择主联系人（保留对象）

这种方式比“拖拽到详情页合并”更符合常见桌面管理心智。

### 移动端建议
不优先支持复杂手工合并。

### 未来方案
可增加“让 Agent 帮我整理”入口，让用户通过自然语言完成联系人清洗与合并。

---

## 12. 与 MessageHub 的关系

Users & Agents 与 MessageHub 高度耦合，但二者职责不同。

### Users & Agents 更关注
- 实体结构
- 导入与来源
- Binding 管理
- 分组与删除
- 资料展示与整理
- Agent 管理入口

### MessageHub 更关注
- 对话
- 好友请求
- 会话状态
- 消息过滤
- 单次添加与即时互动

### 协同建议
- 左侧列表风格保持一致；
- 用户详情悬浮卡、实体卡片等尽量做成系统通用组件；
- 同一实体在两处的名称、头像、状态应保持一致。

---

## 13. 技术与交互约束

### 13.1 应用可用性约束
当前版本下，通常只允许第一个 Owner 用户安装应用。  
后续新增用户默认使用已安装应用，而不承担应用安装入口。

### 13.2 同空间用户通信
同一系统空间中的两个用户可通过 MessageHub 直接通信。  
这类通信更多是系统内部数据操作，不一定产生传统意义上的外部网络行为。

### 13.3 多账号切换约束
当前账号 Session/Cookie 偏全局，不适合频繁快速切号。  
如果需要同时操作两个账号，更适合使用两个浏览器环境。

### 13.4 DID Document 修改成本
Document 未来可能位于空间外、链上或其他可信载体，因此修改不应被设计成“无成本的轻编辑”。

## 14. 原型设计建议

### 14.1 首版原型应覆盖的页面

至少包含以下页面：

1. Users & Agents 首页
2. Self 详情页
3. Agent 详情页
4. 本空间用户详情页
5. 联系人详情页
6. 群组详情页
7. 新建本空间用户向导
8. Binding 管理弹层/子页
9. 导入联系人流程
10. 联系人来源总览页
11. 多选合并确认弹层（桌面端）

### 14.2 首版原型必须体现的状态

- 空状态（无联系人、无绑定）
- 导入中 / 导入完成
- 同步来源状态
- DID 单向关系 / 双向关系
- 可编辑的 Info 与需向导修改的 Document
- Agent 已绑定 / 未绑定
- 群组：集合型 / 实体型
- 自建组 / 加入组

### 14.3 视觉层级建议

- **最高层级**：自己、Agent、自建实体型组
- **中层级**：本空间用户
- **普通层级**：联系人、加入的组、导入对象
- **辅助层级**：来源、标签、状态标识、后缀

## 15. 未来演进方向

### 15.1 安全联合登录
支持已有一级 DID 用户无密码泄露地加入新组织/系统空间。

### 15.2 联系人可信绑定公开化
允许用户在 Profile 中公开自己的外部可达方式，并逐步建立可信验证机制。

### 15.3 AI 辅助联系人整理
包括：

- 自动识别重复联系人；
- 推荐合并；
- 自动清理无效分组；
- 通过自然语言完成整理任务。

### 15.4 手机号 / 邮箱 -> DID 发现
降低 DID 使用门槛，帮助用户从传统联系方式平滑进入 DID 世界。

### 15.5 移动端轻量化管理
移动端优先保留查看、搜索、导入与基础整理，不承载复杂批量操作。

## 16. 关键开放问题

1. “添加联系人”入口是否完全迁移到 MessageHub？
2. 搜索框最终放在左侧顶部还是底部悬浮区？
3. 外部 Binding 的可信验证机制如何设计？
4. 联系人自动合并的阈值、解释性与回滚机制如何定义？
5. 实体型组与集合型组在视觉上如何做到“易懂但不混淆”？
6. 加入的群组是否在左侧始终单独分区？
7. Document 修改涉及成本时，用户支付与确认路径如何设计？

## 17. 最终产品判断

Users & Agents 不应被设计成一个“后台管理页”，而应被设计成：

- 一个面向普通用户的实体管理中心；
- 一个将 **用户、Agent、联系人、群组、DID、Binding** 统一起来的系统级入口；
- 一个既服务当前 BuckyOS 使用体验，又为未来 AI Native 身份与协作模型铺路的基础组件。

它的核心价值不在于“把对象列出来”，而在于帮助用户逐步理解：  
**在 BuckyOS 里，用户与 Agent 都是可被管理、可被连接、可被赋能的实体。**
